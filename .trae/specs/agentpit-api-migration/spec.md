# AgentPit API 迁移规范文档

## 1. 项目概览

**项目名称**：AgentPit API 迁移
**项目类型**：前端 API 集成项目
**迁移目标**：从 Mock 数据向真实 API 迁移，保持系统稳定性和功能完整性

## 2. 背景与上下文

AgentPit 项目目前使用 11 个 Mock 数据文件模拟后端 API 响应，所有业务模块都依赖这些 Mock 数据。随着项目的发展，需要集成真实的后端 API 以提供更真实的用户体验和数据处理能力。

### 当前状态
- 项目使用 Vue3 + TypeScript + Pinia 技术栈
- 11 个 Mock 数据文件覆盖所有业务模块
- 未建立 API 服务层架构
- 缺乏统一的错误处理和数据缓存机制

### 迁移目标
- 保持 Store 层接口不变，减少对现有组件的影响
- 实现渐进式迁移，支持回退到 Mock 数据
- 建立统一的 API 服务层架构
- 实现完善的错误处理和数据缓存机制

## 3. 功能需求

### 3.1 API 服务层架构
- **FR-1**：创建完整的 API 服务层目录结构
- **FR-2**：实现 HTTP 客户端封装，支持 GET、POST、PUT、PATCH、DELETE 方法
- **FR-3**：实现统一的错误处理机制
- **FR-4**：实现数据缓存管理
- **FR-5**：支持环境变量配置，可切换 Mock/真实 API

### 3.2 模块迁移
- **FR-6**：迁移 P0 模块（Chat、Monetization、Sphinx、Home）
- **FR-7**：迁移 P1 模块（Marketplace、Social、Collaboration、Memory）
- **FR-8**：迁移 P2 模块（Customize、Lifestyle、Settings）

### 3.3 质量保障
- **FR-9**：实现端到端测试
- **FR-10**：实现性能优化
- **FR-11**：完善 API 文档

## 4. 非功能需求

### 4.1 性能要求
- **NFR-1**：API 响应时间 ≤ 1000ms
- **NFR-2**：系统稳定性 ≥ 99.9%
- **NFR-3**：错误率 ≤ 1%

### 4.2 可靠性要求
- **NFR-4**：支持网络错误重试机制
- **NFR-5**：实现数据缓存，减少重复请求
- **NFR-6**：支持回退到 Mock 数据

### 4.3 可维护性要求
- **NFR-7**：代码结构清晰，易于维护
- **NFR-8**：完善的 TypeScript 类型定义
- **NFR-9**：详细的 API 文档

## 5. 技术约束

### 5.1 技术栈
- **前端框架**：Vue 3.x
- **状态管理**：Pinia 2.x
- **HTTP 客户端**：原生 fetch API + 轻量封装
- **类型系统**：TypeScript 5.x
- **构建工具**：Vite 6.x

### 5.2 依赖
- **环境变量**：Vite 环境变量管理
- **错误处理**：自定义错误类
- **缓存**：内存缓存 + localStorage

## 6. 架构设计

### 6.1 目录结构

```
src/services/
├── api/
│   ├── client.ts          # HTTP 客户端封装
│   ├── chat.ts            # 聊天相关 API
│   ├── monetization.ts    # 变现相关 API
│   ├── sphinx.ts          # 建站相关 API
│   ├── home.ts            # 首页相关 API
│   ├── marketplace.ts     # 市场相关 API
│   ├── social.ts          # 社交相关 API
│   ├── collaboration.ts   # 协作相关 API
│   ├── memory.ts          # 记忆相关 API
│   ├── customize.ts       # 定制相关 API
│   ├── lifestyle.ts       # 生活服务相关 API
│   └── settings.ts        # 设置相关 API
├── config.ts              # API 配置
├── errors.ts              # 错误处理
├── cache.ts               # 缓存管理
└── index.ts               # 导出入口
```

### 6.2 数据流向

```
组件 → Store → API 服务 → 后端 API
```

### 6.3 错误处理流程

1. API 服务捕获错误
2. 转换为统一的错误类型
3. Store 层处理错误并更新状态
4. 组件层显示错误提示

## 7. 迁移策略

### 7.1 渐进式迁移
- 保持 Mock 数据作为回退选项
- 通过环境变量控制是否使用真实 API
- 分阶段迁移，确保每个阶段的功能正常

### 7.2 测试策略
- 单元测试：测试 API 服务层
- 集成测试：测试 Store 与 API 服务的集成
- 端到端测试：测试完整的用户流程

### 7.3 回退机制
- 当 API 迁移出现问题时，可通过设置 `VITE_USE_MOCK_API=true` 切换回 Mock 数据
- 保留所有 Mock 数据文件，确保回退能力

## 9. 风险评估

### 9.1 主要风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| API 规格变更 | 中 | 高 | 与后端团队保持密切沟通，使用 API 版本控制 |
| 性能下降 | 低 | 中 | 实现数据缓存，优化请求策略 |
| 错误处理不完善 | 中 | 中 | 充分测试边界情况，完善错误提示 |
| 迁移时间超期 | 中 | 低 | 分阶段迁移，保持 Mock 数据作为回退选项 |

### 9.2 风险应对
- 建立与后端团队的定期沟通机制
- 实现完善的监控和预警系统
- 保持 Mock 数据作为应急回退选项
- 分阶段迁移，确保每个阶段的质量
