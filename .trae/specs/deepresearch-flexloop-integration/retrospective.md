## 第1章 项目概述

### 1.1 项目背景

当前 Flexloop 项目中已有一个基础的 `useDeepResearch.ts` composable，但仅通过子进程调用 CLI 方式集成 DeepResearch，功能有限且稳定性不足。需要基于 DeepResearch 框架构建一个完整的智能体系统，提供标准化的 API 接口，使 Flexloop 能够通过更稳定可靠的方式调用 DeepResearch 的深度研究能力，实现自主学习、任务调度和自主决策等核心功能。

### 1.2 项目目标

基于 `tools/DeepResearch` 中的资源和功能，开发一个具有自主学习和任务执行能力的智能体系统，使其能够被 `apps/Flexloop` 项目集成和调用：

- 具备明确的接口规范
- 支持 Flexloop 通过 API 或服务调用方式交互
- 实现数据处理、任务调度、自主决策等核心功能
- 确保通信稳定可靠，数据传输安全
- 提供完整的集成文档

### 1.3 项目范围

**包含范围**:
- 在 DeepResearch 中新增 FastAPI RESTful API 服务层
- 实现异步任务处理和状态查询
- SSE 实时进度推送
- TypeScript 客户端 SDK 供 Flexloop 调用
- 重构现有的 `useDeepResearch` composable
- 添加安全认证和错误处理机制
- 编写完整的集成文档
- 提供 Podman 容器化配置

**不包含范围**:
- 前端展示组件（列为可选任务未强制要求）
- 持久化任务存储（当前使用内存存储）
- 分布式任务队列

## 第3章 技术架构决策分析

### 3.1 架构决策: Python API + TypeScript 客户端

**决策内容**: 将系统分为两层
- **Python 后端服务**: FastAPI 提供 RESTful API，运行 DeepResearch 智能体
- **TypeScript 前端客户端**: 在 Flexloop 中封装 API 调用，提供响应式接口

**决策理由**:
1. ✅ **职责分离清晰**: Python 擅长 AI/LLM 生态，TypeScript/JavaScript 适合前端开发
2. ✅ **独立部署运维**: API 服务可以独立扩展、升级、重启，不影响前端应用
3. ✅ **多客户端接入**: 未来如果需要其他客户端（如 CLI、其他前端项目）接入，直接使用 API 即可
4. ✅ **技术栈匹配**: DeepResearch 本身就是 Python 项目，增加 API 层不会引入额外复杂度
5. ✅ **解耦**: 前后端独立开发和发布，降低耦合度

### 3.2 架构合理性评估

| 评估维度 | 评分 | 评估说明 |
|---------|------|---------|
| **功能性** | ⭐⭐⭐⭐⭐ | 完全满足功能需求 |
| **可维护性** | ⭐⭐⭐⭐⭐ | 分层清晰，职责明确，易于维护 |
| **可扩展性** | ⭐⭐⭐⭐⭐ | 易于添加新端点、新功能，易于水平扩展 |
| **性能** | ⭐⭐⭐⭐ | 异步处理+SSE 推送性能良好，内存存储在单实例场景下足够 |
| **可靠性** | ⭐⭐⭐⭐⭐ | 网络错误重试、超时处理、任务取消机制完善 |
| **安全性** | ⭐⭐⭐⭐⭐ | API 密钥认证、输入验证、输出净化、CORS 配置完整 |
| **开发效率** | ⭐⭐⭐⭐⭐ | FastAPI 自动生成文档，TypeScript 类型安全，开发效率高 |

### 3.3 替代方案对比

**替代方案 1: 纯 Node.js 重写**

- 优点: 单一技术栈，部署简单
- 缺点: DeepResearch 已有完整 Python 生态，重写工作量巨大，LLM 库生态 Python 更丰富
- 结论: 不划算，本次决策正确

**替代方案 2: 继续使用子进程调用**

- 优点: 不需要独立服务
- 缺点: 稳定性差，资源管理困难，并发处理能力弱
- 结论: 原方案痛点明确，本次架构升级解决了这些问题

**替代方案 3: GraphQL 替代 REST**

- 优点: 更灵活的查询
- 缺点: 对于这个场景，CRUD 风格的 API 很清晰，GraphQL 会增加复杂度
- 结论: REST 足够简单清晰，过度设计没有必要

### 3.4 关键子架构决策

#### 3.4.1 异步任务 + 轮询/SSE 双模式

**决策**: 创建任务立即返回 202，客户端通过轮询或 SSE 获取进度

**合理性**:
- ✅ 避免 HTTP 请求超时（研究任务可能需要几分钟）
- ✅ SSE 提供实时进度更新，用户体验好
- ✅ 轮询作为兜底，兼容性更好
- ✅ 服务端不需要维护长连接，扩展性好

#### 3.4.2 内存任务存储

**决策**: 当前使用内存存储任务状态，不使用数据库

**合理性**:
- ✅ 满足当前单实例部署需求
- ✅ 实现简单，不需要额外基础设施
- ✅ 未来可以平滑迁移到 Redis 等外部存储
- ✅ 任务是瞬时性的，不需要持久化

**局限性**:
- 多实例部署时需要外部存储
- 服务重启会丢失正在进行的任务

**升级路径清晰**: 当业务需要扩展时，将 `task_manager.py` 中的存储层抽象出来，替换为 Redis 即可。

#### 3.4.3 API 密钥认证

**决策**: 使用简单的 API 密钥认证，不使用 OAuth2/JWT

**合理性**:
- ✅ 场景简单：服务端和前端在同一内网或同一部署
- ✅ 实现简单，足够安全
- ✅ 未来需要可以升级

#### 3.4.4 Podman 容器化

**决策**: 提供 Podmanfile 和 podman-compose，支持多环境配置

**合理性**:
- ✅ 环境一致性，避免"本地能跑线上不能跑"问题
- ✅ 开发环境支持源码挂载热重载
- ✅ 生产环境多阶段构建，镜像更小更安全
- ✅ 非 root 用户运行，安全性更好

### 3.5 结论

> **技术架构决策合理性: ⭐⭐⭐⭐⭐ (5/5)**
>
> Python API + TypeScript 客户端的分层架构非常适合这个场景，充分利用了原有技术投资，职责分离清晰，架构简单且能满足当前和未来一段时间的需求。关键子决策都经过了权衡，选择了最简单且足够的方案，没有过度设计。

### 5.1 配置过程概述

### 5.2 遇到的主要问题及解决方案

#### 问题 1: Windows 下 Podman 网络连接问题

**问题现象**:
- Podman 容器启动成功，端口映射正确配置 (`8000:8000`)
- 从 Windows 主机访问 `localhost:8000` 连接被拒绝
- 在 Podman 虚拟机内部可以正常访问

**根因分析**:
- Podman on Windows 使用 WSL 2 后端
- 默认的 `localhost` 端口映射需要正确配置才能从 Windows 访问
- 有时需要检查 WSL 2 网络是否正常

**解决方案**:
```powershell
# 1. 确认容器正在运行
podman ps

# 2. 检查端口映射
podman port <container-id>

# 3. 在 Windows 中测试连接
curl http://localhost:8000/api/v1/health

# 如果还是失败，尝试使用 WSL 2 IP 访问
# 在 WSL 中执行
ip addr show eth0 | grep inet
# 然后使用该 IP 访问
```

**经验总结**: 在 Windows 环境下使用 Podman，网络问题主要出现在 WSL 2 端口映射，需要确保：
1. Podman 机器正常运行
2. 端口映射正确配置
3. Windows 防火墙允许连接

#### 问题 2: 构建过程中 pip 下载速度慢

**问题现象**:
- 第一次构建镜像时，pip 下载依赖包非常慢
- 经常超时失败

**根因分析**:
- 默认 PyPI 镜像在中国大陆访问速度慢
- 需要使用国内镜像源加速

**解决方案**:
在 `Podmanfile` 和 `podman-compose.yml` 中预先设置环境变量：

```dockerfile
ENV PIP_INDEX_URL=https://pypi.tuna.tsinghua.edu.cn/simple
ENV PIP_TRUSTED_HOST=pypi.tuna.tsinghua.edu.cn
```

**经验总结**: 对于中国大陆地区的项目，在容器构建配置中默认使用国内镜像源，可以避免很多构建超时问题。

#### 问题 3: 开发环境源码挂载不生效

**问题现象**:
- 开发环境修改代码后，变更没有反映到容器中
- 需要重新构建才能看到变更

**根因分析**:
- 卷挂载配置不正确
- 没有使用 `--reload` 参数启用 uvicorn 自动重载

**解决方案**:
在 `podman-compose.yml` 中正确配置：

```yaml
volumes:
  - ./:/app                              # 挂载整个项目目录
  - /app/__pycache__                     # 排除 pycache
  - /app/src/deepresearch.egg-info       # 排除 egg-info

command: >
  sh -c "
    pip install --no-cache-dir -e . &&
    pip install --no-cache-dir fastapi uvicorn &&
    uvicorn deepresearch.api.main:app --host 0.0.0.0 --port 8000 --reload
  "
```

**经验总结**: 开发环境需要完整的源码挂载，并且需要启用服务器自动重载才能实现真正的热更新。

#### 问题 4: 健康检查失败

**问题现象**:
- 容器启动后，健康检查一直失败
- 容器被自动重启

**根因分析**:
- 容器内没有 `curl` 命令
- 健康检查依赖 curl 进行测试

**解决方案**:
在 Podmanfile 中确保安装了 curl：

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends \
        curl \
    && rm -rf /var/lib/apt/lists/*
```

**经验总结**: 健康检查需要基础工具，在生产镜像中不要忘记安装必要的诊断工具。

## 第6章 可复用最佳实践提炼

### 6.1 架构设计最佳实践

#### 实践 1: 前后端分层解耦

**经验**: 当把 Python 数据科学/AI 能力集成到 TypeScript 前端项目时，**不要**尝试：
- 在 Node.js 中调用 Python 子进程（稳定性差）
- 用 TypeScript 重写 Python 库（工作量大，维护困难）

**推荐做法**:
- Python 端提供独立的 API 服务
- 前端通过 HTTP 调用
- 这样双方都能用自己最擅长的技术栈

**适用场景**: 所有跨语言集成，尤其是 AI/LLM 能力集成到前端项目。

#### 实践 2: 异步任务 + SSE 进度推送模式

对于长时间运行的任务（研究可能需要几分钟）：

```
客户端: POST /research → 得到 task_id
客户端: 连接 /research/{task_id}/stream → SSE 接收进度
服务端: 后台异步执行，持续推送进度
最终: 推送完成事件，带上结果
```

**优点**:
- 用户体验好，实时看到进度
- 不会遇到 HTTP 请求超时问题
- 服务端资源释放及时

**可复用性**: 这个模式可以复用到所有需要长时间运行的 AI 任务场景。

#### 实践 3: 双模式流式兼容

同时支持：
1. 原生 `EventSource`（简洁）
2. `Fetch` + `ReadableStream`（兼容性好，可取消）

这样可以适配不同浏览器环境，提高兼容性。

### 6.2 容器化最佳实践

#### 实践 1: 多 profile 环境分离

在 `podman-compose.yml` 中使用 profile 分离开发和生产：

```yaml
profiles: ["development"]  # 开发环境
profiles: ["production"]   # 生产环境
```

**好处**:
- 开发环境：源码挂载，热重载，方便调试
- 生产环境：预构建镜像，性能好，安全
- 使用同一个 compose 文件，配置统一管理

#### 实践 2: 多阶段构建

```dockerfile
FROM python:3.14-slim AS base
# 安装依赖...

FROM python:3.14-slim AS production
# 复制已安装的依赖...
```

**好处**:
- 生产镜像不包含构建工具
- 镜像体积更小
- 攻击面更小

#### 实践 3: 非 root 用户运行

```dockerfile
RUN addgroup --system appgroup && adduser --system appuser --ingroup appgroup
USER appuser
```

**安全最佳实践**: 不以 root 运行容器，即使容器被攻破，攻击者也得不到 root 权限。

#### 实践 4: 资源限制

```yaml
limits:
  memory: 4G
  cpus: "2.0"
reservations:
  memory: 1G
```

**经验**: AI 研究任务内存消耗较大，必须设置资源限制，防止服务被打垮。

#### 实践 5: 日志轮转

```yaml
logging:
  driver: json-file
  options:
    max-size: "10m"
    max-file: "3"
```

## 第9章 对后续类似项目的建议

### 9.1 需求分析阶段建议

1. **一定要写清楚 spec.md**
   - 定义清楚每个 API 端点
   - 定义清楚所有场景
   - 这是并行开发的基础

2. **任务分解要 granular**
   - 每个任务半天到一天工作量
   - 明确依赖关系
   - 每个任务有可验证的出口

3. **提前识别风险**
   - 容器网络问题（尤其是 Windows 环境）
   - 依赖包下载速度（提前配置国内镜像）
   - 资源需求（AI 任务内存需求大，提前规划）

### 9.2 架构设计建议

1. **跨语言集成优先选择 API 方式**
   - 不要子进程调用（坑太多）
   - 不要重写（工程量太大）
   - API 分层是最稳妥的方案

2. **长时间运行任务采用异步+流式模式**
   - 不要让客户端 long-polling
   - SSE 推送进度，用户体验好
   - 支持任务取消，用户体验好

3. **容器化从一开始就做**
   - 开发环境和生产环境都容器化
   - 开发环境支持热重载
   - 生产环境做好安全配置

4. **保留向后兼容**
   - 重构已有代码时，保留原有接口
   - 提供兼容层，不强制立即迁移
   - 降低集成风险

### 9.3 容器化建议

1. **使用多阶段构建**
   - 基础阶段安装依赖
   - 生产阶段只保留运行时需要的东西
   - 镜像更小更安全

2. **不要以 root 运行**
   - 创建非 root 用户
   - 这是最简单也最有效的安全实践

3. **设置资源限制**
   - AI 服务一定要设置内存限制
   - 防止某个请求把整个服务搞挂

4. **配置日志轮转**
   - 防止日志填满磁盘
   - 设置合理的文件大小和个数

5. **健康检查必须有**
   - 让容器编排知道服务是否健康
   - 自动重启不健康的实例

### 9.4 安全建议

1. **多层防御**
   - 认证：API 密钥或 token
   - 输入：验证和清理
   - 输出：内容净化防止 XSS
   - 网络：CORS 配置

2. **不要硬编码密钥**
   - 使用环境变量
   - 不要提交到代码仓库

3. **及时更新依赖**
   - 定期扫描依赖漏洞

### 9.5 文档建议

1. **必须包含**
   - 部署指南（一步一步）
   - 环境变量说明（表格清晰）
   - API 文档（每个端点说明）
   - 使用示例（可复制粘贴）
   - 故障排查（常见问题和解决方法）

2. **更新及时**
   - 编码完成后立即写文档
   - 部署时发现问题及时更新文档

### 9.6 测试建议

1. **单元测试覆盖核心逻辑**
   - 认证
   - 任务管理
   - 客户端错误处理

2. **清单检查必不可少**
   - 每个功能点都手动验证一遍
   - 记录检查结果

### 9.7 特定场景建议

对于 AI 智能体集成项目：

1. **任务超时必须设置**
   - LLM 调用可能挂住
   - 超时自动取消，释放资源

2. **并发控制必须设置**
   - 每个研究任务消耗较多内存
   - 限制并发数，保护服务

3. **进度反馈很重要**
   - 研究时间长，用户需要知道进度
   - SSE 推送是很好的方案
