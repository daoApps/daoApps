## 🎯 复盘机制总览

本机制包含**两种复盘模式**：

| 模式 | 触发方式 | 频率 | 适用场景 | 文档示例 |
|------|---------|------|---------|---------|
| **定期复盘** | Phase 完成 / 周期性 | 每周至少1次 / Phase 结束时 | 进度回顾、质量评估、计划调整 | `phase7-review.md` |
| **不定期复盘** | 事件驱动 / 即时触发 | 按需即时 | 技术卡点、功能障碍、里程碑达成 | `fix-ts-errors-20260410.md` |

---

## ⚡ 不定期复盘机制（重点）

### 一、触发条件（必须立即组织）

#### 🔴 高优先级触发（必须在 2 小时内启动）

| 触发类型 | 具体条件 | 示例 | 响应时间 |
|---------|---------|------|---------|
| **技术卡点** | 开发过程中遇到无法推进的技术问题 | TS 编译错误 >50 个且持续 >30 分钟；构建失败无法解决；运行时崩溃阻塞性 bug | ⚡ **立即** |
| **功能实现障碍** | 核心功能无法按预期实现或存在重大缺陷 | P0 功能缺失关键特性；需求理解出现重大偏差；技术方案不可行需重新设计 | ⚡ **2 小时内** |
| **阶段性任务完成** | 完成一个重要的里程碑或阶段性目标 | Phase 全部完成；TypeScript 零错误达成；测试全量通过；版本发布成功 | ✅ **24 小时内** |

#### 🟡 中优先级触发（应在当天启动）

| 触发类型 | 具体条件 | 示例 | 响应时间 |
|---------|---------|------|---------|
| **质量门禁未通过** | ESLint/测试/性能等质量检查失败 | ESLint 错误 >20 个；测试覆盖率不达标；Lighthouse 分数 < 80 | 📅 **当天** |
| **重大代码重构** | 对系统架构或核心模块进行大规模修改 | Store 重构影响 10+ 组件；路由系统重新设计；状态管理方案变更 | 📅 **当天** |
| **外部依赖升级** | 核心依赖版本升级带来 breaking change | Vue3 升级到新主版本；Vite 重构；Pinia API 变更 | 📅 **当天** |

#### 🟢 低优先级触发（可在本周内完成）

| 触发类型 | 具体条件 | 示例 | 响应时间 |
|---------|---------|------|---------|
| **用户要求** | 用户明确要求进行复盘 | 用户输入 "复盘"；用户要求分析某个具体问题 | 👤 **按用户要求** |
| **团队决策** | 团队成员认为有必要进行复盘 | 发现重复出现的错误模式；希望总结某类问题的解决方案 | 👥 **团队协商** |

### 二、不需要复盘的情况 ❌

以下情况**不需要**启动正式复盘流程：

- ✅ 日常小 bug 修复（< 3 个文件变更，< 10 行代码改动）
- ✅ 纯文档更新或注释修改
- ✅ 依赖版本小版本升级（patch/minor 版本，无 breaking change）
- ✅ 代码格式化或 lint 自动修复
- ✅ 配置文件微调（不涉及架构变更）

## 📝 复盘执行流程（不定期复盘）

### 流程图

```
触发事件发生
    ↓
[Step 1] 快速评估（5 分钟）
    ├─ 是否满足触发条件？ → 否 → 记录日志，继续开发
    └─ 是 → 确定复盘级别（🔴高/🟡中/🟢低）
         ↓
[Step 2] 组织复盘会议（15-30 分钟）
    ├─ 参与人员：相关开发者 + 受影响模块负责人
    ├─ 准备材料：错误日志、代码 diff、测试结果
    └─ 会议形式：线上/线下均可，建议屏幕共享
         ↓
[Step 3] 问题分析（核心环节，20-40 分钟）
    ├─ 问题具体表现（What）
    ├─ 影响范围（Who/Where）
    ├─ 根因分析 RCA（Why）← 使用 5 Whys 方法
    ├─ 解决方案过程（How）
    └─ 耗时统计（When/How long）
         ↓
[Step 4] 经验教训提取（10-15 分钟）
    ├─ ✅ 做得好的方面
    ├─ ⚠️ 需要改进的方面
    ├─ 💡 最佳实践总结
    └─ 🚫 避免的陷阱
         ↓
[Step 5] 制定行动计划（10 分钟）
    ├─ 立即行动（24h 内）+ 责任人
    ├─ 短期行动（本周内）+ 责任人
    └─ 中期行动（本迭代内）+ 责任人
         ↓
[Step 6] 形成书面文档（复盘后 2 小时内）
    ├─ 使用 _template.md 模板
    ├─ 存储至 .trae/reviews/
    └─ 通知相关团队成员
         ↓
[Step 7] 行动跟踪（持续）
    ├─ 立即行动项：24h 内验证开始执行
    ├─ 短期行动项：下次复盘时验证
    └─ 最佳实践：沉淀到项目 wiki 或规范文档
```

### 各阶段时间要求

| 步骤 | 最长时间 | 说明 |
|------|---------|------|
| Step 1: 快速评估 | 5 分钟 | 判断是否需要复盘 |
| Step 2: 组织会议 | 15-30 分钟 | 包括准备和召开 |
| Step 3: 问题分析 | 20-40 分钟 | 这是核心环节，不能省略 |
| Step 4: 经验教训 | 10-15 分钟 | 提取可复用知识 |
| Step 5: 行动计划 | 10 分钟 | 必须明确责任人和截止时间 |
| Step 6: 书面文档 | 2 小时内 | 必须在复盘结束后 2h 内完成 |
| Step 7: 行动跟踪 | 持续进行 | 直到所有行动项关闭 |

**总耗时**: 约 **1-2 小时**（不含后续跟踪）

## 📄 复盘文档必需内容

### 每次复盘必须包含的 6 大板块：

#### 1️⃣ 基本信息（4 项必填）
- [ ] 阶段/事件名称
- [ ] 时间范围
- [ ] 参与人员
- [ ] 复盘日期 + 触发原因

#### 2️⃣ 问题具体表现（详细描述）
```
✅ 必须包含：
- 问题描述（What happened）
- 错误信息/日志截图（Evidence）
- 复现步骤（Reproduction steps）
- 发现时间和发现人
```

#### 3️⃣ 影响范围分析（量化评估）
```
✅ 必须包含：
- 受影响的模块/组件列表
- 受影响的用户/功能点
- 影响程度评估（P0-P3）
- 临时规避方案（如有）
```

#### 4️⃣ 根因分析 RCA（使用 5 Whys 方法）
```
✅ 必须包含：
- 直接原因（Immediate cause）
- 根本原因（Root cause）— 为什么会发生直接原因？
- 贡献因素（Contributing factors）
- RCA 方法说明（使用了什么分析方法）
```

**示例 - TypeScript 334 错误的 RCA**:
```
问题：334 个 TypeScript 编译错误
├─ 直接原因：多个语法和类型错误
│  ├─ enum 语法不被 erasableSyntaxOnly 支持 (3 errors)
│  ├─ 循环类型引用 Required<T,K> (1 error)
│  └─ .find() 返回 undefined 未处理 (~50 errors)
├─ 根本原因：
│  ├─ tsconfig 配置过于严格（noUncheckedIndexedAccess 产生 35+ 噪音错误）
│  └─ React→Vue3 迁移时未充分适配 TypeScript 严格模式
└─ 贡献因素：
   ├─ Mock 数据假设值一定存在（实际可能 undefined）
   └─ 组件间接口契约未对齐（Store 属性名不一致）
```

#### 5️⃣ 解决方案实施过程（可复现）
```
✅ 必须包含：
- 方案选择过程（考虑了哪些方案？为什么选这个？）
- 实施步骤（Step-by-step，足够详细让他人复现）
- 代码变更清单（Files changed, LOC changed）
- 效果验证方法（如何确认问题已解决）
- 耗时统计（总共花了多长时间）
```

#### 6️⃣ 经验教训 + 预防措施（可复用）
```
✅ 必须包含：
- ✅ 做得好的方面（2-5 条，具体到行动）
- ⚠️ 需要改进的方面（2-5 条，可执行的改进建议）
- 💡 最佳实践总结（2-5 条，其他项目也能参考）
- 🚫 避免的陷阱（2-5 条，明确的禁止事项）
- 🛡️ 预防措施（如何避免同类问题再次发生）
```

## 🎯 改进行动计划规范

### 行动项三级分类

| 级别 | 时间范围 | 定义 | 示例 |
|------|---------|------|------|
| **立即行动** | 24 小时内 | 必须马上做的事情，通常是修复或紧急优化 | 修复剩余的 5 个 TS 错误；更新 tasks.md 复选框 |
| **短期行动** | 本周内 | 本周可以完成的改进项 | 为新模块编写单元测试；完善错误处理逻辑 |
| **中期行动 | 本迭代/版本内 | 当前版本发布前需要完成的 | 建立完整的回归测试矩阵；优化构建配置 |

### 每个行动项必须包含的字段

```markdown
| # | 改进项 | 具体行动内容 | 责任人 | 截止时间 | 验收标准 | 状态 |
|---|--------|-------------|--------|----------|---------|------|
| 1 | 修复 TS 错误 | 运行 vue-tsc -b 并修复所有错误 | Developer A | 2026-04-10 18:00 | Exit Code 0 | 🔄 进行中 |
```

**关键字段说明**:
- **责任人**: 必须是具体的个人或角色，不能是"大家"或"团队"
- **截止时间**: 必须是具体日期时间，不能是"尽快"或"有空时"
- **验收标准**: 必须可量化验证（如"Exit Code 0"、"覆盖率 ≥ 80%"）
- **状态**: 🔄 进行中 / ⏳ 待开始 / ✅ 已完成 / ❌ 已取消

## 📊 复盘成果应用

### 成果如何指导后续开发

| 成果类型 | 应用场景 | 具体做法 |
|---------|---------|---------|
| **根因分析报告** | 遇到类似问题时快速定位 | 将常见错误模式录入 FAQ 或知识库 |
| **最佳实践清单** | 新模块开发时遵循 | 在 spec.md 或 coding-guidelines 中引用 |
| **预防措施检查表** | Code Review 时对照检查 | 创建 pre-commit checklist |
| **行动计划跟踪** | 确保改进落地 | 每次复盘时回顾上次的行动项完成情况 |

### 团队能力提升路径

```
第 1 次复盘 → 建立复盘习惯，学会 RCA 方法
    ↓
第 2-3 次复盘 → 积累最佳实践，形成团队知识库
    ↓
第 4-5 次复盘 → 预防同类问题，减少重复犯错
    ↓
持续进行 → 提升整体开发效率，形成质量文化
```

## ✅ 复盘质量检查清单

每次复盘完成后，使用此清单自检：

### 文档完整性检查
- [ ] 包含全部 6 大必需板块
- [ ] 每个问题都有 RCA 分析（至少 3 层 Why）
- [ ] 所有行动项都有明确的责任人和截止时间
- [ ] 包含量化数据（错误数、耗时、影响范围等）
- [ ] 经验教训具有可复用性（不是仅针对本次问题的具体描述）

### 及时性检查
- [ ] 🔴 高优先级触发：复盘在 **2 小时内** 启动
- [ ] 🟡 中优先级触发：复盘在 **当天** 启动
- [ ] 🟢 低优先级触发：复盘在 **本周内** 完成
- [ ] 书面文档在复盘会议结束后 **2 小时内** 完成

### 实用性检查
- [ ] 行动项都是可执行的（不是空泛的"加强xx"）
- [ ] 预防措施可以在未来工作中实际应用
- [ ] 其他团队成员阅读后能够理解问题和解决方案
- [ ] 文档存储位置正确（`.trae/reviews/` 目录下）
