### 决策模式分析

本次复盘覆盖的多个任务中，决策模式呈现以下特点：

1. **需求明确型任务**（变现系统、社交系统、单元测试）：
   - spec已经定义清楚所有需求和范围
   - 按计划顺序执行即可，决策点少
   - 主要决策集中在实现细节上

2. **需求探索型任务**（品牌重命名）：
   - 初始需求模糊，需要逐步探索
   - **渐进式聚焦**决策模式效果很好
   - 先发散再收敛，最后让用户决策

3. **外部依赖型任务**（服务器部署）：
   - 遇到外部环境问题时，及时停下来寻求用户协助是正确决策
   - 不要在不可控因素上浪费太多时间

### 最佳决策实践

| 实践 | 效果 |
|------|------|
| **spec先行**：开发前先写好spec、tasks、checklist | 范围清晰，不遗漏需求 ✅ |
| **渐进式聚焦**：处理模糊需求时先发散再收敛 | 用户满意度高 ✅ |

### 问题详细分析

#### **P1: WebFetch 无法获取动态网页完整内容**

**问题描述**：
```
使用 WebFetch 工具获取网页，只返回了 HTML 的 meta 标签部分，没有实际的正文内容
```

**根因分析**：
- WebFetch 可能不支持 JavaScript 动态加载的内容
- 目标网页使用 React 渲染，内容是客户端渲染的

**解决方案**：
- 切换到 integrated_browser 工具
- 使用 browser_navigate + browser_snapshot 获取完整内容

**结果**：✅ 成功获取完整内容

**经验教训**：
- 对于动态渲染网页，优先使用浏览器工具
- WebFetch 适合静态网页，对SPA支持不好

---

#### **P2: wc -w 统计中文字数不准确**

**问题描述**：
```
wc -w 命令统计结果显示文章只有270字，但实际内容远不止这些
```

**根因分析**：
- `wc -w` 按空格分词，中文不使用空格分隔
- 所以统计结果远低于实际字数

**解决方案**：
- 使用 `wc -c` 统计字符数来间接验证长度
- 中文字符每个字占3字节（UTF-8），可以估算字数

**结果**：✅ 9667字符 ≈ 1600汉字，符合要求

**经验教训**：
- 中文统计字数不要用 `wc -w`
- 使用字符数作为验证标准更可靠

---

#### **P3: 服务器无法连接 Let's Encrypt API**

**问题描述**：
```
certbot申请证书时，无法连接到 acme-v02.api.letsencrypt.org
```

**根因分析**：
- 服务器所在网络环境可能有防火墙限制
- 或者DNS解析问题，无法访问Let's Encrypt API
- 这是**环境问题**，不是配置问题

**解决方案**：
- 需要用户检查服务器网络配置
- 或者使用其他证书颁发机构（如阿里云SSL证书）
- 或者配置端口转发让服务器能够访问API

**当前状态**：⟳ 等待用户协助处理

**经验教训**：
- 外部网络问题提前预判
- 遇到网络问题尽早寻求用户协助
- 不要反复重试浪费时间

---

#### **P4: RunCommand 参数序列化错误**

**问题描述**：
```
invalid params: deserialize params error: invalid type: string "True", expected a boolean
```

**根因分析**：
- 在 JSON 参数中，`blocking` 和 `requires_approval` 需要布尔类型
- 错误地传递了字符串 `"True"` 而不是布尔值 `true`

**解决方案**：
- 正确写法：`"blocking": true` （没有引号）
- 错误写法：`"blocking": "True"`

**结果**：✅ 根因已找到，后续不会再犯

**经验教训**：
- 仔细阅读工具参数定义
- 注意参数类型要求
- 错误信息明确指出问题后，不要重复同样的错误

---

#### **P5: 目录被占用无法移动**

**问题描述**：
```
Move-Item 失败：Cannot move item because the item at ... is in use
```

**根因分析**：
- 当前终端的工作目录就在要移动的目录里面
- PowerShell 不允许移动当前正在使用的目录

**解决方案**：
- 先切换到上级目录再执行移动
- 或者先复制到新位置，验证无误后再删除旧目录
- 或者在文件资源管理器中手动操作

**结果**：✅ 解决方案已明确，等待后续执行

---

#### **P6: Recharts 图表组件导入错误**

**问题描述**：
```
不能正确导入 PieChart, Cell 等组件
```

**根因分析**：
- 导入路径写法错误

**解决方案**：
- 正确写法：
  ```typescript
  import { PieChart, Pie, Cell, ResponsiveContainer } from 'recharts'
  ```
- 检查项目中其他图表组件的导入方式，保持一致

**结果**：✅ 快速解决

---

#### **P7: Tailwind CSS 响应式断点覆盖问题**

**问题描述**：
- 移动端样式在桌面端没有正确覆盖
- grid 列数设置不正确

**根因分析**：
- 断点顺序不正确，应该从移动端到桌面端升序编写
- `md:` 断点需要放在基础样式之后

**解决方案**：
- 正确顺序：`grid grid-cols-1 md:grid-cols-2 lg:grid-cols-4`
- 错误顺序：`lg:grid-cols-4 grid grid-cols-1` （顺序不对）

**结果**：✅ 修正后正常工作

---

#### **P8: Vitest 测试中 Pinia store mock 配置问题**

**问题描述**：
- 测试组件中使用 store 时，mock 配置不正确
- 无法正确监听 store 方法调用

**根因分析**：
- mock 方式不对，vi.mock() 需要正确配置

**解决方案**：
```typescript
vi.mock('@/stores', () => ({
  useAppStore: () => ({
    toggleDarkMode: vi.fn(),
    // ...
  })
}))
```

**结果**：✅ 参考已有测试文件找到正确写法

## 9. 经验方法

### 成功要素

1. **spec驱动开发**：每个功能开发任务都先写spec、tasks、checklist，范围清晰，不遗漏需求
2. **组件化拆分**：按功能拆分组件，每个组件只做一件事，单一职责原则贯彻到位
3. **渐进式需求探索**：处理模糊需求时，采用发散-收敛-再聚焦流程，帮助用户明确真实需求
4. **遵循现有规范**：代码风格、命名方式、目录结构都与现有项目保持一致，降低维护成本
5. **及时经验沉淀**：定期进行系统性复盘，将隐性经验转化为显性文档，持续改进

### 可复用方法论

#### **方法论1：spec驱动开发工作流**
```
1. 理解需求 → 编写 spec.md（目标、需求、验收标准）
2. 分解任务 → 编写 tasks.md（按优先级分解，标记依赖）
3. 验证清单 → 编写 checklist.md（每一项功能的验收检查点）
4. 按任务执行 → 逐个完成，勾选checklist
5. 验证验收 → 检查所有checklist都完成
```
**适用场景**：功能开发任务，需求明确时效果最好

---

#### **方法论2：模糊需求探索工作流（品牌命名类）**
```
1. 理解约束 → 明确来源、风格、字数等要求
2. 第一轮发散 → 提供多个方向的候选，每个附带出处和解读
3. 用户反馈 → 缩小范围，调整方向
4. 第二轮聚焦 → 在缩小范围内再次提供候选
5. 深度挖掘 → 如果用户指定特定方向，深入该方向
6. 用户确认 → 分析用户提出的方案，给予专业评价
7. 最终决策 → 用户确定最终方案
```
**适用场景**：品牌命名、产品选型等需要用户参与决策的探索性任务

---

#### **方法论3：多组件模块开发工作流**
```
1. 定义数据结构 → 先写types和mock数据
2. 开发独立组件 → 从无依赖的基础组件开始
3. 逐个组件推进 → 完成一个再开始下一个
4. 最后开发主页面 → 整合所有子组件
5. 统一样式调优 → 确保所有组件风格一致
6. 响应式适配 → 检查移动端/平板/桌面端
```
**适用场景**：包含多个组件的新功能模块开发

### 最佳实践

#### 开发实践

1. **先写spec再编码**：避免做着做着偏离需求，减少返工
2. **参考现有代码风格**：新项目先看几个现有组件，模仿其写法，保持一致
3. **TypeScript 类型先定义再实现**：先写interface，再写组件逻辑，减少bug
4. **响应式从移动端开始**：先写移动端样式，再用md: lg: 断点增强
5. **测试编写参考已有测试**：复制现有测试的mock配置，避免踩坑

#### 问题解决实践

1. **先看错误信息**：错误信息通常已经指明问题，仔细阅读
2. **找不到问题时，搜索项目中类似场景的做法**：大多数问题项目中已经有解决方案
3. **外部环境问题尽早求助**：不要在不可控问题上浪费时间
4. **工具参数问题对照文档仔细检查**：类型错误通常是格式问题

#### 协作实践

1. **模糊需求多轮沟通**：不要指望一次就能猜对用户想要什么
2. **spec写完后让用户确认**：避免理解偏差，返工成本高
3. **遇到问题主动提供多个解决方案**：让用户做选择题而不是问答题
4. **定期复盘沉淀经验**：系统化整理经验，比零散记忆更可靠

### 知识图谱：常见问题速查

| 问题 | 根因 | 解决方案 |
|------|------|----------|
| WebFetch获取动态网页失败 | WebFetch不支持JS渲染 | 使用 integrated_browser 工具 |
| wc -w 统计中文字数不准 | 中文不按空格分词 | 使用 wc -c 统计字符数 |
| RunCommand 布尔参数报错 | 传递了字符串"true"而不是布尔true | 改为 `"blocking": true` 去掉引号 |
| PowerShell无法移动当前目录 | 当前工作目录就在目标目录内 | 先cd到上级目录再移动 |
| Tailwind响应式不生效 | 断点顺序不对 | 从小到大写：基础 → md → lg |
| 服务器无法连接Let's Encrypt | 网络防火墙限制 | 检查网络配置，或使用其他CA |

### 避免的陷阱

- ❌ 不要不写spec直接开始编码，容易遗漏需求
- ❌ 不要忽视参数类型，JSON对类型要求严格
- ❌ 不要在不可控的外部问题上反复重试浪费时间
- ❌ 不要违背项目现有代码风格，增加维护成本
- ❌ 不要低估多组件任务的工作量，留出缓冲时间
- ❌ 不要跳过单元测试，覆盖率保证未来重构安全
